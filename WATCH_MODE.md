## Quick Start

### 1. Create Initial Archive

Before starting watch mode, create an initial archive:

```bash
chronicon archive \
  --urls https://discourse.example.com \
  --output-dir ./archives \
  --formats html,markdown,github
```

### 2. Start Watch Mode

```bash
# Foreground mode (recommended for testing)
chronicon watch --output-dir ./archives

# Background mode (Ctrl+C to stop)
chronicon watch --output-dir ./archives &

# Check status
chronicon watch status --output-dir ./archives

# Stop daemon
chronicon watch stop --output-dir ./archives
```

## Features

### Core Features

- **Automatic Updates**: Polls for new/modified posts at configurable intervals (default: 10 minutes)
- **Error Recovery**: Exponential backoff on API errors with configurable retry limits
- **Multi-Format Export**: Updates HTML, Markdown, and GitHub exports
- **Process Management**: PID/lock files prevent concurrent runs
- **Graceful Shutdown**: Handles SIGTERM/SIGINT signals properly
- **Status Tracking**: JSON status file with metrics and history

### Git Integration

- **Auto-Commit**: Automatically commit changes after each update
- **Custom Messages**: Template-based commit messages with variables
- **Auto-Push**: Optional push to remote repositories
- **Safety Checks**: Won't commit if working tree has conflicts

### Health Monitoring

- **HTTP Health Checks**: Expose `/health` and `/metrics` endpoints
- **Docker Ready**: Health checks compatible with container orchestration
- **Systemd Integration**: Ready-to-use systemd service file

## Configuration

### Configuration File

Create `.chronicon.toml` in your project directory or `~/.chronicon.toml`:

```toml
[continuous]
# Polling interval in minutes
polling_interval_minutes = 10

# Max consecutive errors before stopping daemon
max_consecutive_errors = 5

# Error backoff multiplier (2.0 = double wait time each error)
error_backoff_multiplier = 2.0

[continuous.git]
# Enable git integration
enabled = false

# Auto-commit changes after each update
auto_commit = true

# Commit on each update (false = batch commits)
commit_on_each_update = true

# Push to remote after commit
push_to_remote = false

# Git remote and branch
remote_name = "origin"
branch = "main"

# Commit message template
# Variables: {new_posts}, {modified_posts}, {topics}, {timestamp}
commit_message_template = "chore: update archive - {new_posts} new, {modified_posts} modified, {topics} topics"
```

### Command Line Options

```bash
# Basic usage
chronicon watch --output-dir ./archives

# Specify formats
chronicon watch --output-dir ./archives --formats html,markdown

# Use custom config
chronicon watch --output-dir ./archives --config ./my-config.toml
```

### Environment Variables

For containerized deployments:

```bash
export CHRONICON_OUTPUT_DIR=/archives
export TZ=UTC
```

## Git Integration

### Setup

1. **Initialize Git Repository**:
   ```bash
   cd archives
   git init
   git add .
   git commit -m "Initial archive"
   git remote add origin git@github.com:your-org/your-repo.git
   ```

2. **Configure Auto-Commit** in `.chronicon.toml`:
   ```toml
   [continuous.git]
   enabled = true
   auto_commit = true
   commit_on_each_update = true
   ```

3. **Enable Auto-Push** (optional):
   ```toml
   [continuous.git]
   push_to_remote = true
   remote_name = "origin"
   branch = "main"
   ```

### Commit Message Templates

Customize commit messages with variables:

```toml
[continuous.git]
commit_message_template = """
chore: archive update

- New posts: {new_posts}
- Modified posts: {modified_posts}
- Topics affected: {topics}
- Timestamp: {timestamp}

🤖 Generated by Chronicon
"""
```

Available variables:
- `{new_posts}` - Number of new posts detected
- `{modified_posts}` - Number of modified posts
- `{topics}` - Number of topics affected
- `{timestamp}` - ISO 8601 timestamp

### SSH Authentication

For push to remote, ensure SSH keys are configured:

```bash
# Generate SSH key (if needed)
ssh-keygen -t ed25519 -C "your_email@example.com"

# Add to SSH agent
eval "$(ssh-agent -s)"
ssh-add ~/.ssh/id_ed25519

# Test connection
ssh -T git@github.com
```

## Deployment Options

### Systemd Service (Linux)

Best for production servers. See [`examples/systemd/README.md`](examples/systemd/README.md) for full details.

1. **Install service**:
   ```bash
   sudo cp examples/systemd/chronicon-watch.service /etc/systemd/system/
   sudo systemctl daemon-reload
   sudo systemctl enable chronicon-watch
   sudo systemctl start chronicon-watch
   ```

2. **Monitor**:
   ```bash
   sudo systemctl status chronicon-watch
   sudo journalctl -u chronicon-watch -f
   ```

### Docker Container

Best for containerized environments. See [`examples/docker/README.md`](examples/docker/README.md) for full details.

1. **Using Docker Compose**:
   ```bash
   cd examples/docker
   docker-compose up -d
   ```

2. **Health checks**:
   ```bash
   curl http://localhost:8080/health
   ```

### Kubernetes

Example deployment:

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: chronicon-watch
spec:
  replicas: 1
  template:
    spec:
      containers:
      - name: chronicon-watch
        image: chronicon-watch:latest
        ports:
        - containerPort: 8080
        livenessProbe:
          httpGet:
            path: /health
            port: 8080
          initialDelaySeconds: 30
          periodSeconds: 60
```

## Monitoring

### Status Command

View current daemon status:

```bash
chronicon watch status --output-dir ./archives
```

Output includes:
- Running status and PID
- Uptime
- Cycle statistics (total, successful, failed)
- Update counts (new posts, modified posts, topics)
- Last/next check times
- Recent cycle history

### HTTP Health Checks

Enable health check server (optional, port 8080 by default):

#### Endpoints

- **`GET /`** - HTML documentation
- **`GET /health`** - JSON health status
- **`GET /metrics`** - JSON detailed metrics

#### Health Check Response

```json
{
  "status": "healthy",
  "healthy": true,
  "running": true,
  "pid": 12345,
  "consecutive_errors": 0,
  "last_check": "2025-11-11T10:00:00",
  "next_check": "2025-11-11T10:10:00"
}
```

Returns:
- `200 OK` if healthy (running and < 3 consecutive errors)
- `503 Service Unavailable` if unhealthy or not running

#### Metrics Response

```json
{
  "status": {
    "running": true,
    "pid": 12345,
    "started_at": "2025-11-11T09:00:00",
    "uptime_hours": 1.5
  },
  "cycles": {
    "total": 6,
    "successful": 6,
    "failed": 0,
    "success_rate": 100.0
  },
  "updates": {
    "total_new_posts": 42,
    "total_modified_posts": 5,
    "total_affected_topics": 15
  }
}
```

### Status File

Watch mode creates `.chronicon-watch-status.json` with:
- Daemon state and PID
- Uptime and cycle counts
- Update statistics
- Recent cycle history (last 50)
- Error information

### Logging

Logs are written using Python's logging module:
- **Console**: Rich-formatted output with colors
- **File**: Structured logs in `chronicon-watch.log`
- **Syslog/Journal**: When running as systemd service

Log levels:
- `INFO` - Normal operations (default)
- `DEBUG` - Detailed debugging (use `--debug` flag)
- `WARNING` - Recoverable errors
- `ERROR` - Serious errors

## Troubleshooting

### Common Issues

#### Daemon Won't Start

**Symptoms**: `chronicon watch` exits immediately

**Solutions**:
1. Check if archive database exists:
   ```bash
   ls -la ./archives/archive.db
   ```

2. Verify no other instance is running:
   ```bash
   chronicon watch status --output-dir ./archives
   cat ./archives/.chronicon-watch.pid
   ```

3. Check for stale lock file:
   ```bash
   rm ./archives/.chronicon-watch.lock
   ```

4. Review logs for errors:
   ```bash
   tail -f ./archives/chronicon-watch.log
   ```

#### Too Many Consecutive Errors

**Symptoms**: Daemon stops after multiple errors

**Causes**:
- Discourse site is down
- Network connectivity issues
- API rate limiting
- Invalid credentials/permissions

**Solutions**:
1. Check site availability:
   ```bash
   curl -I https://discourse.example.com
   ```

2. Increase error threshold in config:
   ```toml
   [continuous]
   max_consecutive_errors = 10  # Default: 5
   ```

3. Increase polling interval to reduce API load:
   ```toml
   [continuous]
   polling_interval_minutes = 20  # Default: 10
   ```

#### Git Push Failures

**Symptoms**: Commits created but not pushed

**Solutions**:
1. Test SSH connection:
   ```bash
   ssh -T git@github.com
   ```

2. Check SSH key permissions:
   ```bash
   chmod 600 ~/.ssh/id_ed25519
   ```

3. Verify git remote:
   ```bash
   cd archives
   git remote -v
   ```

4. Disable push temporarily:
   ```toml
   [continuous.git]
   push_to_remote = false
   ```

#### High Memory/CPU Usage

**Symptoms**: Process consuming excessive resources

**Solutions**:
1. Reduce polling frequency:
   ```toml
   [continuous]
   polling_interval_minutes = 30
   ```

2. Export fewer formats:
   ```bash
   chronicon watch --formats html --output-dir ./archives
   ```

3. Limit systemd resources:
   ```ini
   [Service]
   MemoryMax=1G
   CPUQuota=100%
   ```

### Getting Help

1. **Check logs** first:
   ```bash
   # Foreground mode
   tail -f ./archives/chronicon-watch.log

   # Systemd
   sudo journalctl -u chronicon-watch -n 100

   # Docker
   docker-compose logs --tail=100 chronicon-watch
   ```

2. **Review status**:
   ```bash
   chronicon watch status --output-dir ./archives
   ```

3. **Test manually**:
   ```bash
   # Run single update cycle
   chronicon update --output-dir ./archives
   ```

4. **Enable debug logging**:
   ```bash
   chronicon watch --debug --output-dir ./archives
   ```

5. **Report issues**:
   - GitHub: https://github.com/your-org/chronicon/issues
   - Include logs, status output, and configuration

## Best Practices

### Production Deployments

1. **Use systemd or Docker**: Don't run watch mode in terminal
2. **Enable health checks**: Monitor daemon with `/health` endpoint
3. **Set resource limits**: Prevent runaway resource usage
4. **Use git auto-commit**: Track all changes in version control
5. **Monitor logs**: Set up log aggregation and alerting
6. **Test failover**: Verify daemon restarts after errors

### Security

1. **Run as non-root user**: Create dedicated user for chronicon
2. **Restrict permissions**: Archive directory should be owned by chronicon user
3. **Secure SSH keys**: Use proper permissions (600) and passphrase
4. **Use HTTPS**: Always use https:// URLs for Discourse sites
5. **Review commits**: Audit git commits for sensitive data before pushing

### Performance

1. **Tune polling interval**: Balance freshness vs. API load
2. **Export selective formats**: Only generate formats you need
3. **Use local storage**: Avoid network-mounted archive directories
4. **Monitor API rate limits**: Respect Discourse site's rate limits
5. **Scale vertically first**: Add more CPU/memory before horizontal scaling

## Advanced Usage

### Multiple Forums

Run separate watch instances for multiple forums:

```bash
# Forum 1
chronicon watch --output-dir ./archives/forum1 &

# Forum 2
chronicon watch --output-dir ./archives/forum2 &
```

Or use systemd templates:

```ini
# /etc/systemd/system/chronicon-watch@.service
[Service]
ExecStart=/usr/local/bin/chronicon watch --output-dir /var/lib/chronicon/%i
```

```bash
sudo systemctl enable chronicon-watch@forum1
sudo systemctl enable chronicon-watch@forum2
```

### Custom Healthcheck Port

Set custom port via config or environment:

```toml
[continuous.monitoring]
health_check_port = 8081
```

### Quiet Hours

Pause updates during specific hours (configuration example):

```toml
[continuous]
# Not yet implemented - coming soon
quiet_hours = ["22:00-06:00"]
```

## Migration from Manual Updates

If you're currently running manual updates via cron:

1. **Disable cron job**:
   ```bash
   crontab -e
   # Comment out chronicon update line
   ```

2. **Start watch mode**:
   ```bash
   chronicon watch --output-dir ./archives
   ```

3. **Verify operation** for 24 hours before fully removing cron job

4. **Clean up**: Remove old cron scripts
